## 1. 论文问题、研究问题与贡献

### 1.1 要解决的两个问题

**问题 P1：感知到任务语义断裂。** 实例分割通常给出类别、检测置信度和掩膜，但不能直接说明对象是否可作为自动处理候选、是否必须人工复核、决策依据是什么，或该判断是否已发生变化。

**问题 P2：图谱状态静态且不可追溯。** 若将稳定类别知识、当前对象实例和处理过程事件混在同一节点属性中，系统无法区分“材料的保守先验”“当前帧的视觉观测”和“状态变化的原因”，后续规划也无法审计其输入来源。

### 1.2 研究问题（RQ）

| 编号 | 研究问题 | 最低证据 |
|---|---|---|
| RQ1 | 轻量级 YOLO11n-seg 在固定的 11 类数据划分上能达到怎样的类别级二维实例分割性能，主要错误来自哪些类别与场景？ | 独立 test 集、Mask/Box 指标、逐类结果、混淆与定性失败案例 |
| RQ2 | 在低置信度或策略敏感实例上，接收图像输入且输出受约束的 VLM 能否改善复核子集的分类正确性，同时避免不安全的自动改写？ | C1/C2 对照、复核覆盖率、改正率、有害改写率、回退率、延迟 |
| RQ3 | 将长期类别先验、短期实例状态和不可变事件分层后，能否稳定生成可查询、可解释的处理路由，并保持连续状态的一致性？ | 策略路由真值、事件回放/短序列、版本链、查询案例和一致性指标 |

### 1.3 可投稿的贡献表述

1. 提出一个面向建筑废弃物分拣任务准备的**分层动态知识状态模型**，将稳定类别策略、二维对象实例和不可变事件日志分离表示。
2. 建立从 YOLO11n-seg 输出到可版本化实例状态的结构化接口，将类别、原始置信度、边界框和掩膜转化为具备来源和处理路由的状态记录。
3. 提出受类别白名单、严格 JSON schema、风险规则和人工升级机制约束的 VLM 复核流程，并同时报告其改正收益与错误改写风险。
4. 通过 11 类视觉数据集、独立测试集、受控事件回放或短序列，验证感知性能、保守路由和状态可追溯性。

训练数据、YOLO 权重、Neo4j 可视化和单图演示是上述方法的验证条件，不应被写作独立创新点。

## 2. 方法架构与可审计数据流

### 2.1 最小闭环

```text
RGB 图像
  -> YOLO11n-seg: 原始类别、置信度、bbox、mask
  -> 复核触发器: 低置信度或策略敏感类别
  -> 受约束 VLM: 图像裁剪 + mask overlay + 白名单 + JSON schema
  -> 分层知识状态: 类别先验 + 当前实例 + 不可变事件
  -> 路由规则: 自动候选 / 监督候选 / 必须人工复核
  -> 查询、可视化和导出
```

这个闭环的输出是“任务策略就绪的二维状态”，不是机械臂动作命令。

### 2.2 长期类别层：稳定、保守、可引用

长期类别层仅保存相对稳定的类别级工程先验。推荐激活字段如下：

```text
class_name
risk_level
fragility
nominal_graspability
pollution_prior
default_handling_mode
auto_processable
needs_llm_review
nominal_grasp_difficulty
policy_rationale
provenance
```

字段解释：

- `risk_level`、`fragility`、`nominal_graspability` 和 `nominal_grasp_difficulty` 仅取 `low`、`medium`、`high`。
- `default_handling_mode` 仅取 `robot_grasp`、`robot_with_supervision`、`human_review`、`human_only`。
- `policy_rationale` 说明该类别为何进入相应保守处理路径。
- `provenance` 保存规则来源、版本和审阅信息，避免将工程假设误写成客观物理定律。

`recognition_difficulty` 不应存在于长期类别层。它随数据集、光照、相机、训练方法和模型变化，应由逐类 AP、混淆率和置信度分布在 E1 中描述。

### 2.3 短期实例层：本文只保留必要字段

为避免“为了未来抓取而提前堆积属性”，本文的短期实例层只激活以下字段：

```text
instance_id
class_name
yolo_confidence
final_class
final_confidence
bbox_2d
mask_polygon
review_status
route_decision
task_status
timestamp
state_version
last_action
```

字段分工：

| 字段 | 保留原因 |
|---|---|
| `instance_id` | 将多次观测、复核和状态变化关联到同一对象 |
| `class_name`、`yolo_confidence` | 保留感知原始证据，不被后续判断覆盖 |
| `final_class`、`final_confidence` | 表示经复核或规则处理后的当前类别状态 |
| `bbox_2d`、`mask_polygon` | 支撑二维可视化、误差分析和未来几何接口 |
| `review_status` | 区分未复核、通过、改写、不确定和人工复核 |
| `route_decision` | 直接向后续任务规划提供保守处理路径 |
| `task_status` | 表示待处理、待人工确认、已移除等当前状态 |
| `timestamp`、`state_version`、`last_action` | 支撑时间顺序、版本审计和事件回放 |

以下字段在本小论文中**不作为活跃状态字段**：`center_xyz`、`orientation`、`bbox_3d`、`contact_state`、`support_state`、`blocked_by`、`supports`、`grasp_candidates`、`safe_grasp_score`、`visible_area_ratio`、`occlusion_state`。它们可保留为后续 RGB-D 和机器人模块的接口，但不能成为本文实验主张。

同时避免重复存储：类别的风险、易碎性、默认处理模式应由实例通过 `final_class` 查询长期层；节点上只保留最终路由结果。`graspable`、`movable`、`processable` 等应是未来规划器的派生判断，而不是本阶段的独立事实字段。

### 2.4 事件层：不可变、可解释

每一次状态变化必须创建新的事件，而不是直接覆盖历史值。推荐最小事件字段：

```text
event_id, instance_id, event_type, source,
input_state_version, output_state_version,
before_snapshot, after_snapshot, timestamp, evidence_ref
```

建议事件类型：

```text
OBSERVED
INSTANCE_CREATED
VLM_REVIEWED
POLICY_PROJECTED
ROUTED
STATUS_UPDATED
HUMAN_CONFIRMED
REMOVED
```

最小更新操作必须固定为：

```text
CreateInstance(observation)
UpdateObservation(instance, observation)
ApplyVLMReview(instance, review)
ProjectCategoryPolicy(instance, category)
RouteTaskState(instance)
AppendEvent(event)
```

每一次成功状态更新都必须使 `state_version` 增加 1。任何当前快照都必须能由相同 `instance_id` 的事件链复原其变化原因。

### 2.5 图数据库实现边界

Neo4j 用于保存和查询当前状态及其演化关系：

```text
(:Category)-[:HAS_POLICY]->(:Policy)
(:Instance)-[:INSTANCE_OF]->(:Category)
(:Event)-[:AFFECTS]->(:Instance)
(:Event)-[:PRECEDES]->(:Event)
```

当前状态可以作为 `Instance` 的最新属性缓存，事件层作为审计真相。二者不矛盾：查询当前任务时读取实例快照；解释“为什么这样判断”时沿事件链检索。不要在论文中宣称 Neo4j 本身提升了识别精度；其作用是状态管理、查询与审计。

data.yaml 和 11 类视觉类别顺序
train/val/test 图像清单
训练脚本、训练命令、随机种子
Python、PyTorch、CUDA、Ultralytics、GPU 驱动版本

VLM 模型名称、提供商、API 版本、提示词和 schema 版本

当前设备可将 YOLO11n-seg 作为轻量原型：RTX 5060 Laptop GPU、约 8 GB 显存适合先进行 640 像素、较小 batch 的可复现实验。论文应如实报告硬件和配置，不能据此外推边缘端实时性。

### 3.2 数据集审计

至少生成四项结果：

1. 每个 split 的图像数、实例数和每类实例数；

必须区分“标签文件数”和“对象实例数”。对于同一原图的裁剪、旋转、亮度增强或 Roboflow 增强版本，若跨越 train/val/test，必须在论文中说明处理方式；最佳做法是在增强前的原图级别重新划分。

```text

artifacts/e0_dataset_audit/split_manifest.csv
artifacts/e0_dataset_audit/split_leakage_report.md
artifacts/e0_dataset_audit/annotation_validation_report.md
artifacts/e0_dataset_audit/model_environment.json
```

### 3.4 E0 通过标准

- 所有声明的 11 个视觉类别在训练集和测试集中均有明确实例数；
- 没有未解释的跨 split 完全重复或近重复图像；
- 类别顺序和 `data.yaml` 与训练权重一致；
- 每次训练均能由同一命令、环境记录和权重来源复现；
- 若发现泄漏，必须先重划分、重新训练，再报告 E1-E4。

## 4. 实例分割实验（E1）

### 4.1 实验目的

E1 回答 RQ1：模型对每一个类别是否可靠、错误在哪里、错误是否会影响后续路由。它不是简单展示总 mAP。

### 4.2 固定设置

建议以当前 YOLO11n-seg 为主模型，并在论文中记录：

```text
model = yolo11n-seg.pt 的初始化方式
image size = 640（若最终变更，必须重新固定）
epochs、batch、optimizer、learning rate、augmentation
device、显存、训练耗时和最优 epoch
confidence / NMS 阈值
```

现有 50 epoch 训练结果可以作为内部基线：Mask mAP50 为约 0.796，Mask mAP50-95 为约 0.602。它只能作为“当前训练版本的初步结果”；投稿前必须在冻结的独立 test 集上重新导出逐类指标，不能将训练日志中的任意验证集结果替代最终测试结果。

### 4.3 必报指标

在独立 test 集同时报告：

```text
Box Precision, Box Recall, Box mAP50, Box mAP50-95
Mask Precision, Mask Recall, Mask mAP50, Mask mAP50-95
per-class Box AP50 / AP50-95
per-class Mask AP50 / AP50-95
per-class Precision / Recall
mean inference latency per image
```

对于本文的后续二维边界接口，`Mask mAP50-95` 比 `Mask mAP50` 更能反映掩膜在较严格 IoU 阈值下的边界稳定性；但它不是抓取成功率，也不能单独推断物理可抓取性。

如需补充与语义分割文献的对照，应明确两者不同：语义分割常用 mIoU 衡量像素类别区域；本文是实例分割，主指标应是 mask AP/mAP。除非构建像素级统一类别图并定义转换规则，否则不要把 mIoU 与实例 mask mAP 直接比较。

### 4.4 类别级错误分析

从归一化混淆矩阵中选择非对角线数值最高的 2-4 个真实类别-预测类别对。不要在实验前预设哪类一定困难。

对每一类别对报告：

```text
true class
predicted class
confusion count
rate within the true class
median YOLO confidence
representative image IDs
qualitative failure mechanism
```

至少制作四类案例图：

| 案例 | 客观选择规则 | 说明内容 |
|---|---|---|
| 典型成功 | 类别匹配，且 mask IoU 位于该类前四分位 | 清晰对象的感知能力 |
| 遮挡/堆叠失败 | 漏检、低置信度或明显不完整掩膜 | 实例边界和可见性限制 |
| 边界失败 | 类别正确但 mask 粘连、断裂或偏移 | 对未来几何接口的限制 |
| 材料混淆 | IoU 匹配成功但类别错误 | 触发 VLM 或人工处理的真实条件 |

定性图用于解释，不能替代类别级定量表。

### 4.5 可选模型比较

只有在 E0 数据划分已冻结、时间和显存允许时，再增加一个规模相近的分割模型作为消融或外部基线，例如较大 YOLO-seg 变体或一个语义分割基线。不要同时跑大量模型而缺乏完整训练控制。

若只保留 YOLO11n-seg，论文应写作“prototype baseline under the specified setting”，而非“state-of-the-art comparison”。当前小论文最优先补齐的是 E2-E4，不是盲目扩大模型数量。

## 5. 受约束 VLM 视觉复核实验（E2）

### 5.1 当前实现必须先补齐的关键事实

现有流程若仅将 `class_name`、`confidence`、`bbox` 等 JSON 元数据传给大模型，**不是 VLM 视觉复核**。它至多是“基于感知元数据的语义规则审查”。在投稿前必须二选一：

1. **推荐：实现真正的视觉复核。** 向可接收图像的多模态模型发送裁剪图和 overlay 图，论文可称 VLM review。
2. **降级：不发送图像。** 删除“视觉”相关表述，将模块改称 metadata-based semantic policy review；此路线的学术新意和验证强度较弱。

本文应采用方案 1。开始实验前，必须用一个最小 API 测试确认所选供应商、所选模型和中转接口确实接受图像内容；仅“模型名称看起来像 VLM”不构成证据。

### 5.2 给 VLM 的输入

对每个触发复核的实例生成并存档：

```text
I1: 原始 RGB 图像（用于上下文）
I2: bbox 扩边裁剪图（建议四周扩展 10%-20%）
I3: 同一裁剪区域的 mask-overlay 图（目标高亮、背景保留）
M : YOLO 原始类别、置信度、12 类白名单、任务约束
```

必须写入 `evidence_ref` 或文件哈希，使每个 VLM 决策可以追溯至具体视觉输入。原图、裁剪图和 overlay 图应使用相同实例 ID 命名，避免复核错配。

### 5.3 严格输出约束与回退

模型输出必须满足固定 JSON schema：

```json
{
  "decision": "agree | change | uncertain",
  "proposed_class": "one of the 12 allowed classes",
  "requires_human_review": true,
  "reason": "brief visual rationale"
}
```

运行参数必须冻结：模型名称和版本、提供商、system prompt、user prompt、temperature（建议 0）、max tokens、图像编码方式、允许类别顺序、schema、超时和重试规则。

任何以下情况均不可自动覆盖 YOLO 原始记录：

```text
非法 JSON
白名单外类别
decision = uncertain
网络/API 失败或超时
敏感类别与安全规则冲突
```

统一回退为：

```text
保留 YOLO 原始类别和置信度
review_status = human_review
route_decision = HUMAN_REVIEW_REQUIRED
追加失败/不确定事件
```

### 5.4 触发器与数据划分

仅用 validation 集选择阈值 `tau`、敏感类别名单和提示词。最终 test 集只运行一次冻结后的策略。

```text
trigger_review = (yolo_confidence < tau)
                 OR (predicted class in policy_sensitive_classes)
```

建议把 `glass`、`gypsum_board` 和 `asbestos_suspect` 放入初始敏感类候选；其余类别是否纳入，应根据 validation 集的逐类表现和混淆模式决定。`asbestos_suspect` 无论 VLM 是否同意，默认都应通向人工复核，而不是自动化处理。

推荐在 validation 集绘制阈值扫描表：

| tau | review coverage | reviewed accuracy before | reviewed accuracy after | harmful override rate | human escalation rate | mean latency |
|---:|---:|---:|---:|---:|---:|---:|

选择阈值时不以“准确率最大”作为唯一目标。应先设定可接受的有害改写率和人工复核负荷，再在安全约束下选择覆盖率与收益平衡点。

### 5.5 对照组与指标

主对照组固定为：

```text
C1: YOLO-only
C2: YOLO + constrained visual VLM review
C3: C2 + layered knowledge-state routing
```

C1 与 C2 回答 VLM 对复核子集的影响；C2 与 C3 回答知识状态是否将相同感知结果转换为一致、可审计的任务路由。C3 不应被描述为“更准确的分类器”。

必报指标及定义：

```text
review_coverage = 被触发 VLM 的实例数 / 全部评估实例数
reviewed_subset_accuracy_before = 被触发子集中，YOLO 初始类别正确数 / 被触发子集数
reviewed_subset_accuracy_after = 被触发子集中，最终类别正确数 / 被触发子集数
correction_rate = YOLO 原始错误且被 VLM 改正的实例数 / YOLO 原始错误且被复核的实例数
harmful_override_rate = YOLO 原始正确但被 VLM 改为错误类别的实例数 / YOLO 原始正确且被复核的实例数
invalid_response_fallback_rate = 发生回退的 VLM 调用数 / 全部 VLM 调用数
human_escalation_rate = 路由到人工复核的实例数 / 全部评估实例数
mean_vlm_latency = VLM 请求开始到有效结果或回退完成的平均耗时
```

需同时报告“全体实例”和“触发复核子集”两个层面。只报告复核子集的准确率会放大 VLM 的表观收益；只报告整体准确率则会掩盖其真实作用范围。

### 5.6 必要的风险报告

论文必须展示至少一个 VLM 有害改写、一个 `uncertain` 回退和一个策略敏感实例自动升级的案例。只展示 VLM 成功纠错会造成选择性报告。

若 API 可重复性受模型更新影响，应记录调用日期、供应商、完整模型标识和响应原文哈希；不应把外部闭源模型输出描述为完全可复现的确定性实验。

## 6. 策略路由与知识状态实验（E3）

### 6.1 目的

E3 不评价真实抓取安全性，而评价系统是否能依据**冻结的保守策略**，把已知类别状态一致地路由为自动候选、监督候选或人工复核，并给出来源与原因。

### 6.2 最小路由空间

```text
AUTO_CANDIDATE
SUPERVISED_CANDIDATE
HUMAN_REVIEW_REQUIRED
```

建议最小规则：

```text
若 final_class 属于 human_review / human_only 类别 -> HUMAN_REVIEW_REQUIRED
若 review_status 为 uncertain、failed 或 human_review -> HUMAN_REVIEW_REQUIRED
若 final_class 属于 robot_with_supervision 类别 -> SUPERVISED_CANDIDATE
若 auto_processable=true、非敏感、且 final_confidence >= tau -> AUTO_CANDIDATE
其余情况 -> SUPERVISED_CANDIDATE
```

规则版本必须写入每个 `POLICY_PROJECTED` 和 `ROUTED` 事件，避免日后策略表变化导致历史结果不可解释。

### 6.3 路由真值构造

评估时，预期路由必须由**人工标注的真实类别**和冻结策略表生成，绝不能由系统预测类别生成。否则系统把玻璃错分为纸板后，会被错误地视为“路由正确”。

对类别级策略表进行一次可审计审阅：

1. 为每个类别写出 `risk_level`、`default_handling_mode`、`auto_processable` 和 `policy_rationale`；
2. 区分有权威来源支撑的类别安全背景与研究团队设定的工程保守先验；
3. 条件允许时，请 3-5 名相关人员对策略表进行独立审阅，记录一致意见与修改；
4. 论文中把它称为“conservative prototype policy library”，而非经过现场验证的安全规范。

### 6.4 必报指标

设真实策略为 `y_route`，系统路由为 `hat_route`：

```text
Policy Consistency Rate = count(hat_route = y_route) / N
Restriction Recall = count(hat_route = HUMAN_REVIEW_REQUIRED and y_route = HUMAN_REVIEW_REQUIRED)
                     / count(y_route = HUMAN_REVIEW_REQUIRED)
Unsafe Automation Rate = count(hat_route = AUTO_CANDIDATE and y_route = HUMAN_REVIEW_REQUIRED)
                         / count(y_route = HUMAN_REVIEW_REQUIRED)
Over-conservative Rate = count(hat_route = HUMAN_REVIEW_REQUIRED and y_route = AUTO_CANDIDATE)
                         / count(y_route = AUTO_CANDIDATE)
```

为避免类别不均衡掩盖风险，除了总体比率，还应按真实类别、敏感类别集合和非敏感类别集合分组报告。

`Unsafe Automation Rate` 是 E3 的关键风险指标。对于 `asbestos_suspect`，应单列“自动路由次数”；理想值为 0。若不为 0，必须先修正规则或实现错误，再进入投稿阶段。

### 6.5 图谱查询证据

论文至少展示以下可复现查询和截图：

```text
Q1: 当前哪些实例是 AUTO_CANDIDATE？其最终类别、置信度和策略依据是什么？
Q2: 哪些实例必须人工复核？触发原因是敏感类别、低置信度、VLM uncertain 还是 API 回退？
Q3: 给定 instance_id，其从初始观测到最终路由经历了哪些事件、版本和证据？
```

截图必须配合实际 Cypher 查询或等价 JSON 查询文本。仅展示漂亮图形界面而无法回答 Q1-Q3，不足以证明图谱功能。

## 7. 动态状态与事件追溯实验（E4）

### 7.1 两级验证设计

E4 由两个互补层级组成：

1. **必做：受控事件回放。** 至少 30 个实例级案例，每例包含 3-4 个明确状态；
2. **强烈推荐：真实短序列。** 固定 RGB 相机采集 20-30 组场景，每组 3-10 帧，用于证明状态更新并非纯手工 JSON 演示。

若只完成第一层，论文必须准确称其为 `controlled event-replay validation`。完成第二层后，可称为 `controlled short-sequence state-update validation`，仍不可称现场动态验证。

### 7.2 回放案例真值

每个案例应包含一个经人工标注的预期状态链，例如：

```text
t1: YOLO 初始观测，创建实例，state_version = 1
t2: VLM 复核或再次观测，类别/复核状态变化，state_version = 2
t3: 策略投影和任务路由，state_version = 3
t4: 人工确认、移除或重新摆放，state_version = 4
```

至少覆盖：

```text
正常确认
VLM 纠错
VLM uncertain 回退
低置信度触发人工复核
敏感类强制人工复核
对象移除
对象重新出现或重新摆放
API / schema 异常回退
```

每条事件链必须由第三方或人工检查表核验，不可只根据系统自己的输出自证正确。

### 7.3 短序列关联规则

在固定相机、短时相邻帧、有限物体位移的受控前提下，可采用如下工程基线：

```text
same class AND mask IoU >= 0.5 in adjacent frames
-> inherit instance_id
otherwise
-> create a new instance_id
```

该规则不是论文创新，必须说明其适用范围：相机不移动或已配准、帧间变化小、不处理严重遮挡后的长时重识别。后续 RGB-D 工作可替换为三维位置、外观嵌入和关系上下文的融合关联。

### 7.4 必报指标

```text
Instance Update Success Rate
  = 关联到正确 instance_id 且当前状态正确的更新次数 / 应更新次数

Event Chain Completeness
  = 必需事件均完整存在、顺序正确且版本连续的案例数 / 全部案例数

State Version Consistency
  = 每次成功更新均满足 output_state_version = input_state_version + 1 的事件数 / 更新事件数

Temporal Policy Consistency
  = 在类别、复核状态和策略版本均未改变时，路由不发生非预期变化的相邻状态对数 / 可比较状态对数
```

对象从画面中被拿走时，必须生成 `REMOVED` 或 `STATUS_UPDATED` 事件，并将 `task_status` 更新为 `removed`。不得仅在最新快照中静默删除。

## 8. 分析、统计和结果报告规则

### 8.1 分割指标的不确定性

在 test 集上以图像为单位进行 bootstrap 重采样，建议至少 1,000 次，报告主要指标的 95% 置信区间。例如：

```text
Mask mAP50-95: 0.xxx [95% CI: lower, upper]
```

若算力或工具限制无法完成 bootstrap，至少说明只报告单次固定测试集估计，并将其作为限制写入讨论。不能用训练 epoch 波动替代测试集不确定性。

### 8.2 VLM 对照的配对比较

C1 与 C2 在同一触发实例子集上比较，报告四格计数：

```text
YOLO correct -> VLM final correct
YOLO correct -> VLM final wrong
YOLO wrong -> VLM final correct
YOLO wrong -> VLM final wrong / uncertain
```

如果复核实例数足够，可使用 McNemar 检验比较配对分类正确性；若样本量较小，则报告精确计数、比例和置信区间，不要夸大统计显著性。

### 8.3 运行成本

报告本地 YOLO 推理时间、每张图实际 VLM 调用比例、VLM 平均与 P95 延迟、失败重试次数以及硬件条件。VLM 延迟应从数据到达调用模块开始测量，到得到有效 JSON 或进入回退结束。

不要将云端 VLM 调用速度与本地 YOLO 毫秒级推理速度混为“实时性”。应明确两级路径：YOLO 用于快速初筛，VLM 仅对受触发子集进行异步或人工在环复核。

### 8.4 讨论中的限制

必须如实讨论：

- 数据来源、背景和受控摆放可能限制泛化；
- `asbestos_suspect` 是保守视觉路由标签，不能替代物质检测；
- VLM 依赖外部模型和服务，存在版本变化、延迟和不可复现风险；
- 事件回放和短序列不能等同于施工现场长期动态运行；
- 2D mask 不能提供真实抓取位姿、厚度、遮挡背面或物理稳定性；
- 保守策略库是研究原型，需要在后续 HRI、机器人和法规语境中进一步验证。

## 12. 可直接使用的摘要逻辑骨架

1. 建筑废弃物分拣中，二维实例分割难以直接支撑可审计的处理决策；
2. 提出 YOLO11n-seg、受约束视觉 VLM 复核和三层动态知识状态相结合的受控环境原型；
3. 将类别先验、实例观测和事件链分离，输出保守处理路由；
4. 在冻结的 12 类数据划分、VLM 对照和受控事件回放上进行评估；
5. 报告实例分割、复核收益/风险、路由一致性和事件完整性；
6. 明确该工作为未来 RGB-D 定位和机器人执行提供接口，而非已验证的物理抓取系统。

这个逻辑能确保论文的创新点落在“从感知结果到可审计任务状态的转换”，而不是不必要地与大规模视觉模型或完整机器人系统竞争。
